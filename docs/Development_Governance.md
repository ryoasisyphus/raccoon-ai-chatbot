# Raccoon AI 開發治理手冊 (Development Governance Guidelines)

本文件總結了專案開發過程中的關鍵失誤，並將其轉化為現代軟體工程的治理原則，旨在確保專案長期維護的穩定性與一致性。

## 1. 代碼異動與完整性 (Code Integrity)
- **拒絕盲目覆蓋**: 禁止在未確認現有邏輯的情況下進行大規模檔案重寫。優先使用精確的行號替換，保護既有的資料資產（如 FAQ、API 連結）。
- **結構自檢機制**: 每次修改函式或異動邏輯後，必須檢查大括號 `{}` 的閉合完整性，嚴禁產生語法錯誤 (Syntax Error)。
- **隔離代碼污染**: HTML、CSS 與 JS 必須嚴格解耦，禁止將視覺標記誤植入邏輯腳本中。

## 2. 視覺與體驗一致性 (UI/UX Cohesion)
- **設計系統繼承**: 所有新組件必須遵循 `docs/Premium_UI_UX_Standards.md`。嚴禁使用藍色底線連結或原生瀏覽器按鈕作為最終交付。
- **玻璃擬態強制執行**: 浮層與彈窗必須具備 `backdrop-filter: blur(30px)`，保持視覺語言的統一。

## 3. 交互穩定性規範 (Interaction Stability)
- **事件冒泡治理**: 在開發下拉選單或模態視窗時，必須使用 `event.stopPropagation()` 隔離點擊事件，避免全域監聽器造成的「閃退」或「連鎖觸發」。
- **範圍判定邏輯**: 應使用 `.contains(event.target)` 進行精確的點擊範圍判定，而非依賴不穩定的單一類別匹配。

## 4. 資料驅動防護 (Data Integrity)
- **GAS 串接優先**: 嚴禁在重構過程中移除或降級 Google Sheets 的串接邏輯。
- **本地備援機制**: FAQ 與 PRODUCTS 的本地 fallback 資料必須維持正確的數據結構，以便在網路失敗時提供一致的用戶體驗。

## 5. 發佈與文檔治理 (Release & Documentation Governance)
- **CHANGELOG 強制更新**: **每次將版本推送到 GitHub 之前，必須先更新 `CHANGELOG.md`**。
- **置頂新增模式**: 新的紀錄必須寫在檔案最上方（置頂），嚴禁覆蓋或刪除舊有的歷史紀錄。
- **語言統一規範**: 所有的開發文檔、提交訊息 (Commit Message) 以及 `CHANGELOG.md` 必須使用 **繁體中文** 撰寫，以確保專案一致性。
- **治理精神同步**: 在每次對話開啟或大型任務開始前，應先複習本手冊，確保開發行為不偏離既定精神。

## 6. 版本分支與工作區隔離策略 (Git Flow & Worktree)
- **分支開發原則**: 嚴禁在 `main` 分支進行實驗性或大規模開發。
    - **新功能**: 必須在 `feature/[功能名稱]` 分支進行。
    - **修復/優化**: 必須在 `fix/[問題名稱]` 或 `refine/[優化項目]` 分支進行。
- **主動提議機制**: 針對「高風險變動」（如：重構核心邏輯、更改 CSS 架構、整合新第三方服務），AI 必須主動提議開立分支，獲得使用者核准後方可執行。
- **Worktree 環境隔離**: 對於需要並行處理多個任務或「徹底實驗」的狀態，應主動建議使用 `git worktree` 建立獨立的物理目錄，確保 `main` 的代碼結構不被實驗性代碼汙染。
- **合併流程 (Merge)**：在分支完成開發並通過驗收後，需先更新分支的 `CHANGELOG.md`，最後方可合併回 `main`。
